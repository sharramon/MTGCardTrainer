# MTG Card Trainer

Data-prep pipeline for training a custom YOLOv7 object detector to recognize
specific Magic: The Gathering cards, for use in the
[`Spectacles_ObjectDetect`](../Spectacles_ObjectDetect) SnapML lens (see that
project's `MLSpatializer`/`DepthCacheSpatializer` for how the trained model
gets used to track cards in 3D space and float content above them).

Currently targeting three cards:

- `Cloud, Ex-SOLDIER`
- `Sephiroth, Planet's Heir`
- `Tifa, Martial Artist`

Internally these are referred to by short folder/class names: `Cloud`,
`Sephiroth`, `Tifa` (matching `Cards/<name>/` and the generated `data.yaml`).

## Why this pipeline exists

A YOLO-style detector is a **closed-set** recognizer -- it has to be trained
on the specific cards you want it to know, it can't recognize an arbitrary
card it's never seen. But because a trading card is a flat, rigid, printed
pattern (unlike a 3D object, it always looks like *itself*, just transformed),
we can bootstrap a large, well-annotated training set from a **single
high-res reference image per card**, almost entirely synthetically -- rather
than needing to physically photograph each card hundreds of times in every
lighting condition and angle.

The full data pipeline:

```
   reference scan/photo (1 per card, in Cards/<Name>/)
            |
            v
   mask_cards.py            -- removes white corner artifacts -> processed/
            |
            v
   download_backgrounds.py  -- pulls varied real-world stock photos -> backgrounds/
   generate_procedural_backgrounds.py -- adds cheap solid/gradient/texture filler
            |
            v
   generate_synthetic_dataset.py -- composites cards onto backgrounds with
                                    randomized rotation/perspective/lighting,
                                    auto-generating YOLO bounding-box labels -> output/
            |
            v
   (separately) extract_frames.py + Roboflow annotation -- small real-world
   supplement that closes the synthetic-to-real gap
            |
            v
   Train in Paperspace (YOLOv7) -> export to ONNX -> drop into
   Spectacles_ObjectDetect/Assets/MLModels/ and wire up MLSpatializer
```

## Folder layout

```
Cards/<Name>/<scan>.jpg       Reference card images you provide (1+ per card)
processed/<Name>/<scan>.png   Alpha-masked versions (generated by mask_cards.py)
backgrounds/                  Pool of background images for synthetic compositing
videos/<Name>/*.mp4           Short videos of each physical card (you provide)
frames/<Name>/*.jpg           Frames extracted from those videos (for Roboflow annotation)
output/                       Generated synthetic dataset (images/, labels/, data.yaml)
scripts/                      All pipeline scripts (see below)
```

## Step-by-step

### 1. Add reference images

Drop one clean, high-res photo or scan of each card into `Cards/<Name>/`
(folders for `Cloud`, `Sephiroth`, `Tifa` already exist). An official
high-res card image works great -- the goal is one crisp, well-lit shot per
card; everything else gets generated from it.

> **Note:** if your source image has rounded corners exported onto a
> rectangular JPG, the corner areas are usually filled with solid white
> (JPGs can't store transparency). `mask_cards.py` (next step) strips that
> out automatically -- you don't need to pre-clean the images.

### 2. Mask out the white corners

```
python scripts/mask_cards.py
```

Converts each `Cards/<Name>/*.jpg` into an alpha-masked
`processed/<Name>/*.png` with the white corner triangles made transparent,
so synthetic compositing produces a true rounded-rectangle card silhouette
(matching how a real card looks sitting on a surface) instead of teaching the
model to look for white corner artifacts that don't exist in reality.

### 3. Build a pool of background images

```
python scripts/download_backgrounds.py [count] [width] [height]
python scripts/generate_procedural_backgrounds.py [count] [width] [height]
```

- `download_backgrounds.py` pulls real stock photos from
  [Lorem Picsum](https://picsum.photos) (no API key needed) directly to disk
  -- broad, generic visual variety (rooms, textures, objects, lighting).
- `generate_procedural_backgrounds.py` creates cheap solid-color/gradient/
  noise/texture fillers locally with no network dependency, for extra padding.

Both are purely **generic filler** -- they teach the model to ignore
arbitrary backgrounds. They are not a substitute for the real-world video
supplement in step 5, which is what teaches the model what *your* actual
environment looks like.

### 4. Generate the synthetic dataset

```
python scripts/generate_synthetic_dataset.py [count_per_class] [out_size]
# e.g. python scripts/generate_synthetic_dataset.py 150 640
```

For each sample, the script:
1. Picks a random background and a random card.
2. Randomly scales, rotates, and perspective-warps the card (simulating it
   lying at different angles/tilts, or being held).
3. Alpha-composites it onto the background at a random position (occasionally
   cropped at the frame edge, for realism).
4. Sometimes draws a soft random "occluder" blob over part of the card
   (simulating a finger/hand partially covering it).
5. Applies brightness/contrast/color-cast/blur/noise jitter to the whole
   frame (simulating different lighting and camera conditions).
6. Writes the image plus a YOLO-format label
   (`<class_id> <x_center> <y_center> <width> <height>`, normalized) -- and
   because *we* control the placement, the bounding box is known exactly, so
   **no manual annotation is needed for this synthetic portion**.

Output lands in `output/images/{train,val}/` and `output/labels/{train,val}/`,
with a `data.yaml` listing the 3 classes -- ready to train directly, or to
import into Roboflow alongside the real-world set.

### 5. Capture and extract real-world frames

Record a short video (30-60 seconds) of each physical card -- slowly
rotating/tilting it and moving it across a few different surfaces and
lighting spots in the environment you'll actually use the Lens in. Drop the
recordings into `videos/<Name>/` (e.g. `videos/Cloud/cloud_table.mp4`), then:

```
python scripts/extract_frames.py [frames_per_video] [sharpness_percentile]
# e.g. python scripts/extract_frames.py 25 30
```

This samples frames evenly across each video, filters out the blurriest ones
(via Laplacian-variance sharpness scoring), and writes a manageable,
de-duplicated set to `frames/<Name>/`.

### 6. Annotate the real-world set in Roboflow

Upload `frames/<Name>/*.jpg` to Roboflow and draw bounding boxes around the
card in each frame, labeling each box with the matching class name (`Cloud`,
`Sephiroth`, `Tifa`). This is the only manual-annotation step in the whole
pipeline -- everything synthetic came pre-labeled. Roboflow's model-assisted
labeling tools can speed this up further. Merge/export this set together with
the synthetic `output/` set in YOLOv7 format.

### 7. Train (Paperspace) and export to ONNX

Follow the `Spectacles_ObjectDetect` README's SnapML workflow, pointing at
this combined dataset's `data.yaml`:

```bash
python train.py --data data.yaml --cfg cfg/training/yolov7-tiny.yaml \
  --weights yolov7-tiny.pt --img 224 224 --batch-size 64 --epochs 200 \
  --name detection --device 0 --hyp data/hyp.scratch.tiny.yaml

python export.py --weights runs/train/detection/weights/best.pt --grid \
  --simplify --export-snapml --img-size 224 224 --max-wh 224
```

### 8. Wire the model into the Lens

Drop the resulting `.onnx` into `Spectacles_ObjectDetect/Assets/MLModels/`,
then on the `MLSpatializer` component set `model` to the new asset and
`classLabels` to `["Cloud", "Sephiroth", "Tifa"]` (or the full display names,
as long as the order matches `data.yaml`'s `names` list). Use
`DepthCacheSpatializer` in continuous mode to track detected cards and float
3D content above them as they move.
